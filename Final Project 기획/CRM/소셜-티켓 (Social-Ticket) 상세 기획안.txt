CRM 프로젝트: "소셜-티켓 (Social-Ticket)" 상세 기획안
1. 사용자 시나리오
매니저 (이팀장):

'소셜-티켓'에 로그인하여 '미처리 소셜 멘션' 피드를 확인한다.
"배송이 너무 늦네요. 실망입니다."라는 부정적인 트윗을 발견하고, 즉시 '신규 티켓 생성' 버튼을 누른다.
생성된 티켓을 '최담당'에게 배정한다.
칸반 보드에서 해당 티켓이 '신규' 열에 생성된 것을 확인하고, 처리 과정을 추적한다.
담당자 (최담당):

로그인 후 '내게 할당된 티켓' 목록에서 신규 배송 지연 티켓을 확인한다.
티켓 상세 정보에서 원본 트윗 내용을 보고 상황을 파악한다.
고객에게 별도로 연락하여 문제를 해결한 후, 티켓의 상태를 '처리중'에서 '완료'로 변경하고 "고객과 통화 후 환불 조치 완료"라는 메모를 남긴다.
칸반 보드에서 해당 티켓이 '완료' 열로 이동한 것을 본다.
2. 화면 설계 (UI/UX)
공통: 로그인 페이지
대시보드:
좌측: 지정된 키워드(예: '우리회사')를 포함한 트윗이 실시간으로 수집되는 피드 영역. 각 트윗 옆에 '티켓 생성' 버튼.
우측: 나의 할 일(To-do) 목록과 최근 활동 로그
티켓 관리 (칸반 보드):
'신규', '처리중', '완료' 3개의 열로 구성된 보드.
티켓 카드를 드래그 앤 드롭하여 상태를 변경할 수 있음.
티켓 상세 모달:
카드 클릭 시 나타나는 팝업. 원본 트윗 내용, 고객 정보, 담당자, 상태 변경, 내부 메모 작성 기능 포함.

4. 핵심 로직: 트위터 멘션 수집
구현: Spring Boot의 @Scheduled 어노테이션을 사용하여 주기적인 작업 스케줄러를 만듭니다.
프로세스:
@Scheduled(fixedRate = 300000): 5분마다 아래 작업을 자동으로 실행합니다.
twitter4j와 같은 트위터 API 클라이언트 라이브러리를 사용하여, 사전에 설정된 검색어(예: '우리회사')로 최신 트윗을 검색합니다.
DB에 저장된 마지막 트윗 ID 이후에 작성된 새 트윗들만 가져와 중복을 방지합니다.
가져온 트윗 내용을 TICKET 테이블에 status를 'NEW'로 하여 저장하거나, 별도의 SOCIAL_MENTION 테이블에 저장 후 티켓으로 전환하게 할 수도 있습니다. MVP에서는 바로 TICKET으로 생성하는 것이 간단합니다.
5. 구체화된 1.5개월 (6주) 개발 계획
1주차 (설계/환경): 기획 확정, DB설계, Twitter 개발자 계정 신청 및 API 키 발급.
2주차 (백엔드): 회원 기능, 티켓 CRUD 및 상태 변경 API 개발.
3주차 (백엔드/통합): 트위터 멘션 수집 스케줄러 개발 및 API 연동 테스트.
4주차 (프론트엔드): 로그인, 멘션 피드 대시보드, 칸반 보드 UI(드래그앤드롭 라이브러리 활용) 개발.
5주차 (마무리): API 전체 연동, 티켓 상세 모달 창 구현, 통합 테스트 및 버그 수정.
6주차 (배포/발표): Azure 배포, 최종 테스트 및 발표 자료/시연 준비.
